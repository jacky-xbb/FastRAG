# 废止标准当普通文档处理（撤销「命中必标已作废」）

## 状态

已采纳（2026-06-15）。**反转**原硬约束⑥（废止标准命中时必须显式标注「已作废」）。

## 背景

原设计把废止标准（如 jc 684-1997）当特殊文档：仍入库带 `废止` 状态，但检索/作答全程特殊对待——
`sanitizeFilter` 护栏防漏召回、`formatHits` 给废止块加「（该标准已作废）」、Agent 指令强制答案标注、
web UI 红标、`hasDeprecatedNotice` 判定。评测里还专门有一档 P1 测「答案是否标了已作废」。

实测（test/eval.ts --llm，12 题）发现：这套强制标注**模型会偶发漏标**（案例12 检索层已给废止标记，
最终答案却没带「该标准已作废」）。要把它做稳，得继续加重 Agent 指令、加校验——为一个边缘行为持续投入。

## 决策

**废止标准当普通文档处理。** 照常入库、检索、作答，不再有任何基于「废止」状态的特殊行为：

- 删 `sanitizeFilter` 护栏、`ChunkFilter.状态` 过滤字段；
- `formatHits` 不再给废止块加标注；
- 删 `hasDeprecatedNotice` 与 web UI 红标；
- Agent 指令 / 检索工具去掉废止相关话术与 `status` 参数；
- 评测去掉 P1（废止标注），废止标准的题保留为**普通召回题**。

**保留**：`状态` 元数据字段、入库写状态逻辑（`statusFromFileName`）、`ChunkMeta.状态`。这是数据事实，
库里已入库（删字段要重灌 ~560MB 向量库），保留不碍事，也为将来若想恢复区分留余地。

## 代价 / 已知风险

- **用户可能拿作废标准当现行用**：jc 684-1997 已作废近 30 年，系统不再提示，用户问到时会当有效标准答。
  这是工程语境下的真实风险，决策时已知并接受（owner 拍板，优先简化）。
- 若日后要恢复「已作废」提示，按本 ADR 反向操作即可：状态数据还在，无需重新入库。
